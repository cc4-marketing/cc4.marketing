# /start-2-7 - Service Package from a Real Engagement

## Instructions for Claude

Final lesson of Module 2 - teach the mine → decide → blueprint → derive practice for turning a completed engagement into a sellable service package.

### Lesson Overview

---

**Module 2.7: Service Package from a Real Engagement**

Markit has now run a full campaign cycle for Planerio — brief, content strategy, copy, analytics, competitive research, SEO. That completed engagement is a product waiting to be packaged: "B2B SaaS Launch Campaigns as a Service." Let's package it.

The same practice works on any real client engagement at your own company: the project folder already contains the process, the metrics, and the praise — an agent's job is to mine it and keep the numbers honest.

**Duration:** ~60 minutes

---

### Step 1: Mine the Engagement Corpus

Everything the student built in lessons 2.1–2.6 lives in `exercises/markit/`. That's the reference corpus.

```
I'm productizing Markit's campaign service, using the completed Planerio
engagement as the reference case. Corpus: exercises/markit/

Before writing anything, mine the corpus and save a research file at
exercises/markit/service-package/research-findings.md covering:

1. SERVICE OPERATIONS MODEL - what work was actually done, in what order,
   with what cadence and tooling
2. DELIVERABLES PRODUCED - every brief, strategy, copy asset, report
3. RESULTS & PROOF POINTS - concrete numbers with dates (from the analytics
   and SEO work). Only claims traceable to a file.
4. LESSONS - what was reworked or corrected mid-engagement, and the fix
5. REUSABLE ASSETS - every template/checklist/framework worth productizing

Telegraphic style, cite source filenames, no invented claims.
```

**Teaching point:** the messy documents are the most valuable. A correction mid-campaign becomes a quality-control step in the packaged service. Post-fix process beats marketing adjectives.

### Step 2: Make the Five Decisions

Before any drafting, the student answers five questions — in writing. Have them reply in chat, then save the decisions:

```
Record my packaging decisions at
exercises/markit/service-package/decisions.md:

1. TARGET BUYER: [niche-first, e.g. B2B SaaS startups 10-100 people?]
2. ANONYMIZATION: [name Planerio, or "a B2B SaaS client"? Default: anonymize]
3. DELIVERABLES: [blueprint / case study / landing page / one-pager - which first?]
4. LANGUAGE: [internal blueprint vs external marketing]
5. SERVICE NAME: propose 3-5 names with rationale and let me pick

For pricing: extrapolate tiers from the engagement's scope, but mark every
figure "proposal - confirm with finance before publishing".
```

**Teaching point:** the engagement proves exactly one price/scope point. Everything extrapolated must stay visibly flagged — the agent's job is to surface assumptions, not bury them.

### Step 3: Write the Service Blueprint

```
Write the service blueprint at
exercises/markit/service-package/service-blueprint.md, from the research
file. Sections:

1. Service definition - name, one-line value prop, target customer,
   explicit OUT-of-scope list
2. Service blueprint table - journey stages (Discover → Contract → Onboard
   → Deliver → Review → Renew) × rows (customer actions / what the customer
   sees / what we do backstage / supporting systems)
3. Engagement lifecycle - pilot → retainer → renewal, with the de-risking
   spelled out as a selling point
4. Operating model - roles, capacity unit, cadence - and every LESSON from
   the engagement stated as a rule
5. Deliverables catalog with quality standards
6. Pricing & tiers (flagged as proposals), asset library, anonymized proof
   points

End with unresolved questions for the business owner.
```

**Teaching point:** blueprint-first, brochure-second. The blueprint is the internal source of truth a new strategist could run a second engagement from. Marketing copy derives from it — never the reverse.

### Step 4: Derive the Case Study and Offer Sheet

```
From the blueprint, derive into exercises/markit/service-package/:

1. offer-sheet.md - the single source of truth for all marketing copy:
   positioning, pain points, what-you-get, deliverables, lifecycle,
   pricing, proof points
2. case-study.md - context → challenge → solution → results table →
   quotes → CTA. Anonymized per my decision in Step 2.

QUALITY GATES before showing me anything:
- Anonymization grep on external files for client/people names (if I chose
  anonymize)
- Numbers identical between case study and offer sheet
- Every extrapolated figure listed under "unresolved questions"
```

**Teaching point:** one-way copy flow (blueprint → offer sheet → assets) means a number changes in one place. When a prospect asks "is that real?", every claim traces back through the research file to a source document.

### Step 5: Stretch Goal — Derive a Landing Page

If time remains, derive a single-file HTML landing page from the offer sheet (copy verbatim, one consistent palette) — and have the agent verify no copy drift against `offer-sheet.md` before review.

## Key Teaching Points
- Real engagements are products in disguise - proof, process, pricing already exist in the project folder
- Mine the messy documents: disputes and mid-flight fixes become the most credible processes
- Five decisions (buyer, anonymization, deliverables, language, name) before any writing
- Blueprint → offer sheet → assets: one-way copy flow keeps numbers consistent
- Gates, not vibes: anonymization greps, number checks, flagged assumptions
